# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repo layout (top-level)

The site is hosted as static files at `towergridlock.com` (DigitalOcean). Three HTML files sit at the root, each self-contained:

- **`index.html`** — landing/chooser page that links to both builds. Plain marketing-style page; no game logic.
- **`gemini.html`** — the original 2025 build, generated by Google Gemini. The architecture documented below describes this file.
- **`claude.html`** — a 2026 reimplementation by Claude Code. Different game (grid placement, 15 waves, neon visuals); separate codebase that does not share code with `gemini.html`. Touch this only when the user is asking about the Claude build.

When the user says "the game" without qualifying, ask which build they mean unless context makes it obvious.

## Running the game

There is no build step, package manager, or test suite. The entire game is in `gemini.html`. To run, open `gemini.html` in a browser (e.g. `xdg-open gemini.html`) or serve the directory with any static server (`python3 -m http.server`). Reload the page to pick up edits.

`localStorage` is used for persistence (high score and settings); to reset state during testing, clear the `waveDefense*` keys via DevTools.

## Architecture

`gemini.html` is a single ~3300-line file with three sections:

- **CSS** (lines ~7–791): `<style>` block. Uses CSS variables under `:root` and a 4:3 aspect-ratio `#game-container` sized via `vmin` so the whole UI scales together.
- **HTML body** (lines ~792–962): the canvas plus all UI overlays (counters, tower buttons, tower-management panel, settings menu, game-over/victory screens, custom tooltip).
- **JavaScript** (lines ~963–3297): one inline `<script>` containing the entire game.

### Game loop and coordinate system

`gameLoop(timestamp)` (gemini.html:1478) is the `requestAnimationFrame` driver. Each frame it computes a `step` value scaled by `gameSpeed` (1x/2x/4x/8x) and a clamped `deltaTime`, then runs the fixed sequence:

1. `updateEnemies` → `updateTowers` → `updateBullets` → `updateParticles` → `updateGameEffects` → `checkWaveStatus` → `updateKillStreak`
2. `drawWaypoints` → `drawNextPathIndicator` → `drawBase` → `drawEnemies` → `drawTowers` → `drawBullets` → `drawParticles` → `drawGameEffects` → `drawTowerPreview`

All update/draw functions share global arrays (`enemies`, `towers`, `bullets`, `particles`, `gameEffects`) and global state (`gold`, `health`, `currentWave`, `waveActive`, `paused`, `gameOver`, `gameWon`, `selectedTowerType`, `selectedTowerForManagement`, …). Mutating these globals is how state changes propagate; there is no event bus or framework.

Drawing uses a **logical 800×600 coordinate system** (`GAME_WIDTH` / `GAME_HEIGHT`). The canvas's actual pixel size is set in `resizeCanvas` (gemini.html:1229) and `ctx.scale(scaleFactor, scaleFactor)` is applied at the top of each frame. Mouse coordinates are converted from client-space to logical-space in `handleMouseMove` / `handleMouseClick`. When adding new draw or hit-test code, work in logical 0–800 / 0–600 units.

### Game-data tables

Most tunables are declarative tables near the top of the script — edit these to add content rather than touching update/draw logic:

- `difficultySettings` (gemini.html:989): per-difficulty multipliers + starting gold/health.
- `towerTypes` (gemini.html:1049): each tower has a base stat block plus an `upgrades` array; `maxLevel = upgrades.length + 1`. Upgrade entries can introduce new behaviors (`splashRadius`, `pierceCount`, `numTargets`, `stunChance`, `slowDuration`, `freezeAuraRadius`) that update/draw functions read off the resolved tower instance.
- `enemyTypes` (gemini.html:1059): movement/health multipliers, gold/score reward, plus optional flags like `isFlying`, `armor`, `healAmount`/`healRadius`.
- `paths` (gemini.html:1071): five waypoint arrays. The active path rotates every `wavesPerPath` (5) waves; on rotation, towers are partially refunded and the player rebuilds.
- `waveCompositions` (gemini.html:1082): 25 waves of `{ composition, interval, baseHealth, baseSpeed }`. `totalWaves` must match this length.

### Targeting, bullets, and special effects

- `findTarget` (gemini.html:2140) honors `targetsFlying` / `canTargetGround` and tower range.
- `fireBullet` / `createBulletData` produce bullet records carrying `splashRadius`, `pierceCount`, `slowDuration`, `slowAmount`, `stunChance`, etc., copied from the firing tower so bullets resolve effects independently in `updateBullets`.
- `damageEnemy` (gemini.html:1897) applies armor, slow/stun status, kill rewards, kill streaks (`handleKillStreak`), and bomb-meter charge.
- `activateBomb` (gemini.html:2354) is a screen-wide AOE; the meter fills from kills.

### Audio

`initAudioContext` lazily creates a `AudioContext` on first user interaction (browsers block autoplay). `playSound(type, volMultiplier, duration)` (gemini.html:1134) generates all SFX procedurally with oscillators — there are no audio assets. New sounds are added by extending the `switch` on `type`. All playback respects `isMuted` and `sfxVolume`.

### Settings persistence

`loadSettings` / `saveSettings` (gemini.html:1577 / 1625) read and write these `localStorage` keys:
`waveDefenseHighScore`, `waveDefenseSfxVolume`, `waveDefenseMusicVolume`, `waveDefenseMuted`, `waveDefenseDamageNumbers`, `waveDefenseParticles`, `waveDefenseDifficulty`. Difficulty changes only take effect on `restartGame` (gemini.html:3211), which is why the UI tells the user to restart.

### Pause semantics

There are two pause flags: `paused` (manual pause via the pause button) and `pausedBySettingsMenu` (auto-pause while the settings overlay is open). The game loop still runs while paused, but skips update steps; cosmetic particles continue to fade. Don't conflate the two — `togglePause` early-returns if `pausedBySettingsMenu` is set.
